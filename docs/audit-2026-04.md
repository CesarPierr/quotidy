Audit Quotidy — synthèse
Périmètre : 148 fichiers TS/TSX, ~18k lignes app + ~3.7k lignes de tests, 48 routes API, 16 modèles Prisma, 811 lignes de CSS global. Pas de Server Actions, pas de fetch lib, dark mode hand-rolled, dual-pattern de soumission de formulaires.

Verdict global : base saine et bien testée, mais elle paie la "rétro-compat avec elle-même". Tu as 2 patterns de forms, 2 patterns de modales, 26 routes qui dupliquent le même prologue d'auth+membership, et 5 composants ≥ 400 lignes qui mélangent layout/state/IO. Le terrain est mûr pour passer aux primitives modernes (Server Actions, useOptimistic, Drizzle/Zod-only validation, design tokens dans Tailwind 4 natif) — sans changer le produit, juste en supprimant la moitié du code de plomberie.

2. API — éliminer ~600 lignes de boilerplate
Constat : 26 routes (sur 48) commencent par cette séquence quasi-identique :


const user = await requireUser();
const membership = await db.householdMember.findFirst({ where: { userId, householdId } });
if (!membership) return NextResponse.json({ error: ... }, { status: 404 });
Plus, 22 routes répètent le pattern isDataRequest(request) ? json : redirect.

Recommandation : un seul fichier src/lib/api.ts exposant un wrapper :


export const householdAction = <S extends ZodSchema>(schema: S, handler) =>
  async (req, { params }) => {
    const user = await requireUser();
    const data = schema.parse(await readBody(req));
    const ctx = await loadHouseholdContext(user, data.householdId, { requireManage: false });
    return handler({ user, ctx, data, req });
  };
→ chaque route passe de 60-80 lignes à 10-15 lignes. À combiner avec Server Actions : la moitié des 48 routes peuvent disparaître complètement (déplacées dans src/lib/actions/).

3. Forms — un seul pattern
Aujourd'hui (3 façons de soumettre) :

<form action="/api/auth/login" method="post"> natif (12 occurrences)
<ClientForm action="/api/..."> (4 occurrences)
useFormAction({ action }) (le reste)
Recommandation :

Server Actions partout, importées dans le composant : <form action={completeOccurrence}>
useActionState pour les erreurs, useFormStatus pour les pending states
Suppression de client-form.tsx, use-form-action.ts (114 lignes), de la moitié du middleware CSRF (Next 16 protège Server Actions nativement)
React Hook Form uniquement pour les wizards multi-étapes (task-creation-wizard) — actuellement 4 usages, on peut viser 1
Suppression nette : ~300 lignes, dépendance @hookform/resolvers retirée (déclarée mais inutilisée).

5. CSS / theming — refactor en Tailwind natif
Aujourd'hui : globals.css = 811 lignes. La fin du fichier contient 30 lignes de !important pour patcher le dark mode classe par classe (.bg-white\/60, etc.). C'est l'anti-pattern Tailwind.

Recommandation Tailwind 4 :

Tokens en @theme (déjà partiellement fait, à compléter avec les variantes dark)
Custom properties --coral-500 etc. dérivées : light-dark(coral, coral-dark) (CSS natif depuis Chrome 123)
Utilitaires personnalisés (btn-primary, app-surface) → @utility btn-primary { … } (Tailwind 4 natif)
Plus aucun !important : tout passe par les @variant dark { … } ou light-dark()
next-themes pour le toggle 3-states (light/dark/system) — déjà à 80 % implémenté côté ThemeProvider, juste la persistance côté membre à brancher si tu veux par-utilisateur
Gain : globals.css passe de 811 → ~300 lignes.

6. État, données, perf
a. useOptimistic sur les actions critiques
OccurrenceCard markComplete, skip, reschedule — tous attendent un round-trip serveur. Avec React 19 :


const [optimisticStatus, setStatus] = useOptimistic(occurrence.status);
async function complete() {
  setStatus("completed");
  await completeOccurrenceAction(occurrence.id);
}
→ feedback instantané, rollback automatique en cas d'erreur via revalidatePath.

b. getCurrentHouseholdContext — fetch parallèle déjà bon, mais sur-fetché
Tu charges 30 jours passés + 30 jours futurs à chaque navigation. Trois options :

Streaming + Suspense : envoyer la coquille immédiatement, streamer les actionLogs (pas critiques)
Cache tag : revalidateTag('household:${id}') au lieu d'un router.refresh() global qui invalide tout
Pagination : actionLogs est déjà à take: 40, mais occurrences charge tout en mémoire pour ensuite filtrer en TS — laisser Postgres faire les 3 filtres avec 3 requêtes parallèles ciblées
c. Service Worker
public/sw.js cache aggressivement, oblige à bumper CACHE_VERSION à la main (déjà documenté en pitfall). Serwist automatise tout, y compris le précache du build Next.
